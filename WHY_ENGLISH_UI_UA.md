# ЧОМУ ІНТЕРФЕЙС КОНСТРУКТОРА АНГЛІЙСЬКОЮ?

## TL;DR

**Запитання:** Чому в конструкторі RolesAI.life мітки англійською (`Role Name`, `Archetype`), коли я створюю українську роль?

**Відповідь:** Тому що **структурні мітки** — це мова парсера, а **контент** — це мова твоєї ролі. Як HTML-теги завжди англійські, але текст всередині — будь-якої мови.

---

## АНАЛОГІЯ: HTML

```html
<!-- Структура: англійська -->
<title>Домашній Організатор</title>
<meta name="description" content="Експерт з організації простору">

<!-- Контент: українська -->
<h1>Привіт! Готові трансформувати свій дім?</h1>
<p>Створімо чистий, організований та затишний простір.</p>
```

Ніхто не вимагає перекладати `<title>` на `<назва>`, бо це **синтаксис**, а не контент.

Так само в OpenRML:
```
STEP 1: BASE INFORMATION    ← структурна мітка (EN)
Role Name: Домашній Організатор    ← твій контент (UA)
```

---

## ПРИЧИНА №1: МОВА НАВЧАННЯ LLM

Великі мовні моделі (Claude, ChatGPT, Gemini) **тренувалися переважно на англійських текстах**.

### Статистика навчальних даних:

| Мова | % у навчальному датасеті |
|------|--------------------------|
| Англійська | ~70-85% |
| Китайська | ~5-10% |
| Українська | ~0.1-0.5% |
| Російська | ~1-3% |
| Інші мови | ~5-15% |

**Що це означає:**

1. **Структурні інструкції англійською розуміються краще**
   ```
   ✅ "STEP 1: BASE INFORMATION" — модель чітко розуміє
   ⚠️ "КРОК 1: БАЗОВА ІНФОРМАЦІЯ" — може спрацювати гірше
   ```

2. **Але контент може бути будь-якою мовою**
   ```
   Role Name: Домашній Організатор ✅
   Description: Експерт з організації простору... ✅
   Greeting: Привіт! Готові трансформувати свій дім? ✅
   ```

## ПРИЧИНА №2: МІЖНАРОДНА СУМІСНІСТЬ

OpenRML — **відкритий стандарт**.  
Файл `.orml.txt`, створений в Україні, повинен працювати:
- У Китаї
- У Франції
- У Бразилії
- Через 10 років

### Як це забезпечити?

**Єдина структурна мова** = універсальний парсер.

```
# Українська роль
STEP 1: BASE INFORMATION
Role Name: Домашній Організатор
ROLE_LANG: ua

# Китайська роль
STEP 1: BASE INFORMATION
Role Name: 家庭整理师
ROLE_LANG: zh

# Іспанська роль
STEP 1: BASE INFORMATION
Role Name: Organizador del Hogar
ROLE_LANG: es
```

Один парсер розуміє всі три файли, бо **структура однакова**.

## ПРИЧИНА №3: TECHNICAL DEBT

Уяви, що ми переклали мітки на 50 мов:

```
# Англійська
STEP 1: BASE INFORMATION

# Українська
КРОК 1: БАЗОВА ІНФОРМАЦІЯ

# Китайська
步骤 1：基本信息

# Арабська
الخطوة 1: المعلومات الأساسية
```

**Проблеми:**

1. **50 парсерів замість одного** (або один дуже складний)
2. **Помилки в перекладі** → роль не працює
3. **Версіонування** → кожна мова окремо оновлюється
4. **Підтримка** → 50 разів більше роботи

**Рішення OpenRML:**

✅ Одна структурна мова (EN)  
✅ Контент будь-якою мовою  
✅ LANGUAGE POLICY керує поведінкою

## ЯК ЦЕ ПРАЦЮЄ: LANGUAGE POLICY

```
🌐 LANGUAGE POLICY
──────────────────────────
ROLE_LANG: ua              ← роль за замовчуванням говорить UA
RESPONSE_LANG: auto        ← але може підлаштуватись під user
LANGUAGE_SELECTOR: enabled ← user може вибрати мову в чаті
SUPPORTED_LANGS: ua, en, zh, es, fr
```

### Сценарії:

#### Сценарій 1: Українець створює роль UA

```
Конструктор (UI): англійською
  │
  ├─ Role Name: [input] → вводить "Домашній Організатор"
  ├─ Description: [textarea] → вводить "Експерт з організації..."
  ├─ Greeting: [textarea] → вводить "Привіт! Готові..."
  └─ ROLE_LANG: [radio] → вибирає UA
  
Файл .orml.txt:
  STEP 1: BASE INFORMATION        ← структура (EN)
  Role Name: Домашній Організатор  ← контент (UA)
  
  🌐 LANGUAGE POLICY
  ROLE_LANG: ua
```

**Результат:** Роль говорить українською з українцями.

---

#### Сценарій 2: Англієць використовує українську роль

```
User (англієць): "Hi, I need help organizing my home"

Роль (RESPONSE_LANG: auto):
  - Бачить, що user пише англійською
  - Автоматично відповідає англійською
  
Response: "Hello! I'm here to help you create an organized 
and peaceful home environment. Let's start by assessing..."
```

**Магія:** `RESPONSE_LANG: auto` дозволяє ролі **підлаштовуватись** під мову користувача!

---

#### Сценарій 3: Китаєць хоче українську роль китайською

```
User: 你好，我需要帮助整理我的家

Роль (SUPPORTED_LANGS: ua, en, zh):
  - Бачить китайську
  - Перемикається на zh (якщо є в SUPPORTED_LANGS)
  
Response: "你好！我在这里帮助你创造一个有组织且和平的家居环境..."
```

**Одна роль → багато мов.**

## ПОРІВНЯННЯ: ДО vs ПІСЛЯ

### ❌ Якби ми локалізували структуру:

```
# Український файл
КРОК 1: БАЗОВА ІНФОРМАЦІЯ
Назва ролі: Домашній Організатор

# Англійський файл
STEP 1: BASE INFORMATION
Role Name: Home Organizer

# Китайський файл
步骤 1：基本信息
角色名称：家庭整理师
```

**Проблема:** 3 різні формати → 3 різні парсери → підтримувати неможливо.

---

### ✅ OpenRML підхід:

```
# Всі файли мають однакову структуру
STEP 1: BASE INFORMATION
Role Name: [Домашній Організатор / Home Organizer / 家庭整理师]

🌐 LANGUAGE POLICY
ROLE_LANG: [ua / en / zh]
```

**Переваги:**
- 1 парсер
- 1 стандарт
- 1 документація
- Нескінченна кількість мов контенту

## АНАЛОГІЯ З РЕАЛЬНОГО СВІТУ

### Програмування:

```python
# Синтаксис завжди англійський
def organize_room():
    """Організувати кімнату"""  # ← коментарі UA
    return "Готово!"             # ← рядки UA
```

Ніхто не пише `функція організувати_кімнату():` — це зламає інтерпретатор.

### Міжнародні стандарти:

- **HTTP headers:** завжди англійські (`Content-Type`, `Authorization`)
- **JSON keys:** завжди англійські (`{"name": "Домашній Організатор"}`)
- **XML tags:** завжди англійські (`<title>Домашній Організатор</title>`)

OpenRML — той самий підхід.

## ЩО КОРИСТУВАЧ БАЧИТЬ?

### У конструкторі (UI):

```
┌─────────────────────────────────────┐
│ Role Name *                         │  ← label (EN)
│ ┌─────────────────────────────────┐ │
│ │ Домашній Організатор            │ │  ← ваш input (UA)
│ └─────────────────────────────────┘ │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ Description *                       │  ← label (EN)
│ ┌─────────────────────────────────┐ │
│ │ Експерт з організації простору, │ │  ← ваш input (UA)
│ │ прибирання та підтримки порядку │ │
│ └─────────────────────────────────┘ │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ Role Language *                     │  ← label (EN)
│ ◉ English (EN)  ○ Українська (UA)  │  ← вибір
└─────────────────────────────────────┘
```

### У файлі .orml.txt:

```
STEP 1: BASE INFORMATION               ← структура (EN)
Role Name: Домашній Організатор         ← контент (UA)
Description: Експерт з організації...   ← контент (UA)

🌐 LANGUAGE POLICY                     ← структура (EN)
ROLE_LANG: ua                          ← ваш вибір
```

### У чаті з роллю:

```
User: Привіт! Потрібна допомога з організацією дому
Role: Привіт! Готовий трансформувати твій дім на організований...
      └─────────────────┬──────────────────┘
                 повністю UA!
```

## ВИСНОВОК

**Структура англійською ≠ Контент англійською**

| Що | Мова | Чому |
|----|------|------|
| **Структурні мітки** | EN | Універсальний парсер, міжнародна сумісність |
| **Твій контент** | Будь-яка | Роль говорить твоєю мовою |
| **LANGUAGE POLICY** | Керує поведінкою | Роль підлаштовується під user |

## FAQ

**Q: Чому не зробити UI україською?**  
A: UI можна локалізувати (це в планах). Але **структура файлу** .orml.txt завжди буде англійською з тих самих причин, що HTML-теги залишаються `<title>`, а не `<назва>`.

**Q: Чи можу я створити роль, яка говорить тільки UA?**  
A: Так! Встановіть:
```
ROLE_LANG: ua
RESPONSE_LANG: ua  ← (замість auto)
LANGUAGE_SELECTOR: disabled
```

**Q: Чи працюватиме моя UA роль через 10 років?**  
A: Так, бо структура `.orml.txt` — стандарт, який не залежить від платформи.

**Q: Чи можу я перекласти структурні мітки?**  
A: Технічно так, але це зламає сумісність. Краще залишити структуру EN, а контент зробити будь-якою мовою.

---

**Висновок:**  
Англійська структура + український контент = універсальність + локальність.

Як HTML: теги англійські, текст — твій.
