# OpenRML — Open Role Markup Language

> **Референсна реалізація** · v0.9.0

[English](./README.md) · **Українська**

---

## Бачення

AI-асистенти стали повсюдними. Але спосіб опису ролей для них досі залишається повністю неформальним — розрізнені промпти, скопійовані інструкції, недокументовані домовленості. Немає стандарту. Немає спільного словника. Немає способу валідувати, передавати або розвивати опис ролі між інструментами, командами і часом.

**OpenRML створено, щоб це виправити.**

Open Role Markup Language — структурована, читабельна людиною специфікація для опису особистості AI-асистента, його поведінки, галузі експертизи та патернів взаємодії. Вона розглядає описи ролей як повноцінні артефакти — версіоновані, валідовані, експортовані та інтероперабельні.

Через 3–5 років ми бачимо світ, де:
- Будь-яка AI-платформа приймає `.orml.txt`-опис ролі без перекладу
- Автори ролей публікують, діляться та форкають ролі як код
- Інструменти відповідності перевіряють, що поведінка AI збігається з його оголошеною роллю
- Організації ведуть аудитовані реєстри ролей для всього AI-стеку

Цей репозиторій — **референсна реалізація** цього майбутнього.

---

## Що таке цей репозиторій

```
Цей репозиторій — референсна реалізація OpenRML 1.0.
```

Це не шаблон. Це не boilerplate. Це канонічна демонстрація того, як OpenRML-ролі створюються, валідуються, експортуються та використовуються.

Все тут — доменна модель, формат експорту, система ідентичності, конструктор UI — існує для того, щоб довести: специфікація повна, узгоджена і реалізована.

Майбутні реалізації (CLI-інструменти, розширення VS Code, API-валідатори, нативні мобільні застосунки) вимірюватимуться тим, що визначено тут.

## Архітектура

OpenRML побудований на суворому розділенні відповідальності:

```
src/
├── core/
│   ├── domain/role/          # Канонічний тип Role та перерахування
│   ├── rml/                  # Парсер та експортер — межа специфікації
│   └── services/             # Хешування ідентичності, логіка ліцензій
│
├── application/
│   └── use-cases/            # Бізнес-операції: створити, зберегти, експортувати, імпортувати
│
├── infrastructure/
│   ├── storage/              # Адаптери: IndexedDB, localStorage, session
│   └── template/             # Завантаження та попередній перегляд шаблонів
│
├── data/
│   ├── templates/            # Кураторські шаблони ролей за категоріями
│   └── template-previews.ts  # Шар виявлення
│
└── presentation/
    ├── components/           # Конструктор UI, кроки, живий попередній перегляд
    └── contexts/             # Межі стану React
```

**Ключові архітектурні принципи:**

- **Шар `core/` не має залежностей від UI.** Його можна витягти та використовувати в будь-якому середовищі виконання.
- **Експортер `rml/` — межа специфікації.** Текстовий формат, який він виробляє, І Є стандартом OpenRML.
- **Ідентичність детермінована.** Той самий вміст ролі завжди дає той самий хеш OpenRML Identity.
- **Сховище підключається.** Складений адаптер обирає найкращу доступну стратегію зберігання.

Детальні діаграми та записи архітектурних рішень — у [`architecture/`](./architecture/).

## Приватність і дані

Ваші ролі ніколи не залишають пристрій. Акаунт не потрібен, жоден сервер не отримує ваші дані, телеметрія не збирається. Єдине сховище — IndexedDB вашого браузера, локальна база даних, яка нічого нікуди не передає.

Це усвідомлене архітектурне рішення, а не обмеження. Ролі, створені в RolesAI, часто містять особистий матеріал: цілі, труднощі, контекст ментального здоров'я, сімейну динаміку, кар'єрні тривоги. Такі дані належать тому, хто їх створив — і тільки йому.

**Що і де зберігається:**

| Дані | Сховище | Залишають пристрій? |
|------|---------|---------------------|
| Ваші ролі | IndexedDB (браузер) | Ніколи |
| Налаштування UI (мова, тема) | localStorage (браузер) | Ніколи |
| Кеш шаблонів | sessionStorage (очищається при закритті вкладки) | Ніколи |
| Експортовані `.orml.txt` файли | Ваша файлова система | Лише якщо ви ними поділитесь |

**Експорт — це ваш бекап.** Синхронізації з хмарою немає, відновлення акаунту немає, «відновити з сервера» немає. Якщо ви очистите дані браузера, ролі зникнуть. Експортуйте регулярно — кожен `.orml.txt` файл є повною самодостатньою резервною копією ролі.

**PWA не змінює цю модель.** Встановлення RolesAI як застосунку на екран телефону не створює акаунт і не вмикає серверну взаємодію. Встановлений застосунок використовує те саме локальне сховище браузера, що й вебверсія.

Повний розбір — у [`docs/privacy.md`](./docs/privacy.md).

## Мовна система

**Нова особливість у OpenRML 1.0:** Кожна роль містить Language Policy, яка визначає, як роль спілкується різними мовами.

```
🌐 LANGUAGE POLICY
ROLE_LANG: ua              ← мова ролі за замовчуванням
RESPONSE_LANG: auto        ← роль підлаштовується під користувача
LANGUAGE_SELECTOR: enabled ← користувач може вибрати мову
SUPPORTED_LANGS: ua, en, zh, es, fr
```

**Що це означає:**
- Роль може бути створена українською, але відповідати англійською, якщо користувач пише англійською
- Одна роль працює з користувачами по всьому світу
- Структурні мітки (STEP 1, STEP 2) завжди англійською для універсальності
- Контент (назва ролі, привітання, опис) — будь-якою мовою

## Три рівні використання ролей

OpenRML роль — це не просто промпт. Це **мінімальна когнітивна одиниця**, яка може існувати в трьох формах:

### Рівень 1: Самостійне використання
Вставте `.orml.txt` у ChatGPT, Claude або Gemini — роль активується миттєво.

**Приклад:**
```
User: [вставляє home_organizer_role.orml.txt]
Claude: Привіт! Готовий трансформувати твій дім?
```

### Рівень 2: В агенті
Роль стає "душею" агента з інструментами, пам'яттю та можливістю діяти.

**Приклад:**
```python
agent = Agent(
  role=loadRole('home-organizer.orml.txt'),
  tools=[CalendarTool(), EmailTool()],
  memory=LongTermMemory()
)
```

### Рівень 3: В системі ролей
Множинні ролі, які динамічно перемикаються залежно від контексту.

**Приклад:**
```
User: "Допоможи організувати кухню"
→ активується Home Organizer

User: "Що приготувати на вечерю?"
→ перемикається на Meal Planner
```

Детальніше: [USE_CASES_UA.md](./docs/USE_CASES_UA.md)
